What is it?

our-space is a totally free, distributed, open-source, non-hierarchical, trustless, peer-to-peer social media alternative.

how does it work?

User accounts are identified by an ethereum wallet address. This can be resolved in any number of ways or simply shared directly.

A distributed hash table contains a list of ethereum wallet addresses plus their current root hash, the timestamp for when it was published, and an ethereum signature from the user's address to verify these facts.

Each user has a root hash corresponding to a json object they publish on IPFS and representing the current state of that user's public profile.

The root hash json object contains these things;

  • Etereum address for this user
  • Signature for the data
  • Data
    • Date published
    • IPFS hash for the user's current meta object.
    • IPFS hash for the user's current following object.
    • IPFS hash for the user's current feed object.
    • IPFS hash for the user's current outbox object.

meta object

Each user has a meta object containing whatever they want, but at a minimum containing the following things;

  • Etereum address for this user
  • Signature for the data
  • Data
    • Date published
    • Name
    • Description or bio
    • Photo

Any change to the meta object need only be published to IPFS and then the resulting hash is placed into a new root object for the user. The new root hash is published to whatever means of resolution the user wants to use.

following object

Each user has a following object containing a list of which other users they are following;

  • Etereum address for this user
  • Signature for the data
  • Data
    • Date published
    • Paginated list of ethereum addresses this user is following

If two users are both following each other, then replies posted between them are automatically approved. Replies to your posts from people you are not following go to your inbox for approval before being posted.

feed object

Each user has a feed object containing their posts in this format;

  • Etereum address for this user
  • Signature for the data
  • Data
    • A paginated list of hashes for replies and their children.

what are replies?

A reply is a linked tree item containing the following;

    Reply IPFS Hash: the reply is represented by its IPFS hash.
  • Etereum address for the signatory user
  • Signature for the data
  • Data
    • From: Ethereum address
    • To: Ethereum address
    • Timestamp for when this reply was published.
    • Intended parent hash of this reply. To what are you replying? You can reply to anything that has an IPFS hash. Taking the example of facebook, I could reply to your feed just like you can (Like writing on your wall), or I could reply to your posts, or your comments on a post.
    • Content of the reply. This could also be a hash to the content if it's something large like a video file.

outbox object

Each user has an outbox object linked within their root hash. This contains a list of all the hashes for replies (individually signed) that they would like to send. These are sorted and grouped under their destination ethereum address.

If I am following someone and they post a reply to something I post, then that is automatically approved and added as a child to the node they are replying to.

If a reply is posted to someone who is not following the person psoting the reply, then it goes to an approval inbox.

It seems like it would make sense to have a limit on this so that bobts cant just flood every post. Maybe you can reply to those who are following you, and you can reach the inbox of someone who is a friend of a friend, but beyond that your replies get filtered out.

Here is what the outbox object looks like;

  • Etereum address for this user
  • Signature for the data
  • Data
    • A paginated list of destination users, each containing a list of nodes intended as replies for those users. (This way, when I check my friends' profiles for updates, I can simply look for anything being sent to me and then attach it to the destination node as a child.

resolving root hashes from ethereum addresses

This is the really hard part of the project. Everyone will be constantly updating our root hashses any time anything changes. Getting that new root hash to your followers in a way that is totally distributed and nonhierarchical is a really challenging problem. Distributed naming systems are certainly not a perfectly solved problem yet but lots of people are working on solutions. I think we can use all of these solutions or whichever one we prefer.

One idea I have experimented with during my research for this project has been building a custom DHT network built into the apps people will use to access the network, or built into a distributed service worker on a web page. But there are lots of solutions already in place, and there's no reason not to just use them all.

Call it round-robin resolution. Publish your changes to the Interplanetary Name System, the Ethereum Name System, DNSLink, etc. Just use them all and then whichever one returns an answer first is what we use. This will certainly get better as we move forward, but it works fine for now.

It's also possible to simply create trustless web gateways where you can post your new hash with a signature and they will serve it to whoever asks. Since your followers will be checking the signature anyway, there is no trust involved.
